# QA 用例生成链路介绍

> 面向 QA 组员的使用指南：如何正确使用 AI 生成用例，以及如何共同维护规则。

---

## 用例生成完整链路

```
用户输入需求
    │
    ├─── 文本描述（需求/PRD/口头说明）
    ├─── PR / 分支代码（可选）
    └─── 截图 / 录屏（可选）
    │
    ▼
┌──────────────────────────────────────┐
│  Phase 1: 多源信息采集                  │
│                                      │
│  A. 规则文件（必读）                     │
│     ① qa-rules.md — 通用规范            │
│     ② <module>-rules.md — 模块规则      │
│     ③ 同系列已有用例 — 复用/引用          │
│     ④ requirements/<需求>.md — 需求文档  │
│                                      │
│  B. 代码仓库（可选，按需读取）              │
│     ⑤ PR diff / 分支代码               │
│        → 理解实际实现逻辑               │
│        → 发现需求描述未提及的边界         │
│        → 确认字段校验规则、状态流转        │
│        → 提取 testID / 组件结构          │
└──────────────────────────────────────┘
    │
    ▼
┌──────────────────────────────────────┐
│  Phase 2: 规则维护（自动触发）            │
│                                      │
│  • 需求/模块规则不存在 → 自动创建         │
│  • 已有规则需更新 → 追加变更记录          │
└──────────────────────────────────────┘
    │
    ▼
┌──────────────────────────────────────┐
│  Phase 3: 用例生成                     │
│                                      │
│  • 需求描述 + 代码实现 双视角交叉覆盖      │
│  • 通用维度 + 模块垂直深度               │
│  • 按场景分组，标注优先级                 │
└──────────────────────────────────────┘
    │
    ▼
┌──────────────────────────────────────┐
│  Phase 4: 精简自检 + 输出               │
│                                      │
│  • 去重 / 合并 / 措辞精简               │
│  • 文件落盘到 cases/<模块>/             │
│  • 附加「产品体验建议」板块               │
└──────────────────────────────────────┘
```

## 三个输入源的定位

| 输入源 | 作用 | 谁提供 |
|-------|------|--------|
| **需求描述** | 定义"要测什么" — 功能范围、预期行为 | PM / QA |
| **代码实现** | 补充"实际怎么做的" — 校验逻辑、状态处理、边界条件 | 开发分支 / PR |
| **规则文件** | 约束"怎么写用例" — 格式、优先级、术语统一 | QA 团队共建 |

三者的关系：**需求定义测试范围，代码补全测试深度，规则保证输出质量**。

## 代码辅助用例设计的典型场景

组员可以这样用：

> 给 AI 一个 PR 链接或分支名，让 AI 去读代码，结合需求生成用例

AI 读代码能额外发现的东西：

| 从代码中发现 | 例子 | 对用例的影响 |
|------------|------|------------|
| 校验规则的精确值 | 名称上限是 24 字符（代码里写死的） | 用例直接用精确边界值，不用猜 |
| 状态流转的分支 | 编辑时地址字段 `disabled`，网络切换触发地址清空 | 补充"地址不可编辑"和"切换网络"的验证 |
| 错误处理逻辑 | 重名时 throw 特定错误码 | 用例验证具体的错误提示文案 |
| 平台差异 | 某逻辑只在 `.native.ts` 有，`.web.ts` 缺失 | 标注该用例仅适用特定端 |
| 隐藏功能 | 代码里有"名称历史列表"但需求没提 | 补充名称历史的验证用例 |

## 产品体验建议的定位

| 项 | 说明 |
|----|------|
| **是什么** | QA 在测试过程中从用户视角提出的体验改进建议 |
| **写在哪** | 用例文件末尾，用 `---` 分隔，独立于测试表格 |
| **写不写 Jira** | 不写。仅作为 QA 内部参考和产品沟通素材 |
| **分类维度** | 安全风险 > 操作效率 > 易用性 > 信息层级 |
| **质量门禁** | 每条建议必须过自检（前提是否正确、UI 是否已有、是否暴露安全设计） |

这样用例文档既是测试执行的依据，也是产品改进的参考，但两者职责清晰分离：**表格内是必须验证的测试点，表格外是可选参考的体验建议**。

## QA 需要知道的三件事

### 1. 怎么用

直接告诉 AI：模块、用例名称、需求描述、测试端。AI 会自动读取规则并生成。格式示例：

> 生成测试用例，模块：地址簿，用例名称：地址簿 - 编辑地址。需求：...。测试端：iOS/Android/ext/desktop/web

如果有前置依赖（比如需要先执行添加用例），在需求中说明即可。

### 2. 怎么维护规则

规则分三层，改对应层级的文件：

| 层级 | 文件 | 改什么 |
|------|------|--------|
| **通用规范** | `docs/qa/qa-rules.md` | 格式、优先级定义、允许/禁止词表、通用检测维度 |
| **模块规则** | `docs/qa/rules/<module>-rules.md` | 某模块特有的业务规则、校验逻辑、状态机 |
| **需求文档** | `docs/qa/requirements/<模块>-<功能>.md` | 某功能的需求背景、业务口径、变更记录 |

**核心原则**：规则写在文件里，不写在脑子里。AI 每次生成前会自动读取这些文件，所以规则文件就是唯一事实来源。

### 3. 怎么沉淀经验

- **发现用例生成有误** → 纠正 AI 后，AI 会主动询问是否写入规则文件
- **发现新的通用规律**（如某类校验逻辑多模块复用）→ 补充到 `qa-rules.md` 通用维度
- **发现模块特殊规则**（如 Ripple 的 Tag 只能正整数）→ 补充到对应 `<module>-rules.md`
- **需求变更** → 先更新需求文档，再重新生成用例

这样每次纠正和发现都会沉淀为文档，下次生成自动生效，不依赖任何人的记忆。

---

## 总结

一句话：**需求定义测试范围，代码补全测试深度，规则保证输出质量。**

- AI 不是黑盒生成用例，而是基于项目内规则文件驱动的标准化流程。
- 规则文件是唯一事实来源，团队每个人都可以维护，每次纠正都会沉淀回文件，下次自动生效。
- 除了传统的需求输入，还可以给 AI 提供 PR / 分支代码，让它从实现层面发现需求未提及的边界和校验逻辑。
- 用例文档末尾的「产品体验建议」不进 Jira，仅作为 QA 内部参考和产品沟通素材。

**核心目标：规则写在文件里，不写在脑子里。经验沉淀在项目中，不丢失在聊天记录里。**
