# 转账链规则文档

> 本文档记录不同链的转账相关规则，包括最小转账金额、账户最低余额、Gas 规则等。
> 生成转账测试用例时，必须参考本文档中的规则。

---

## 📋 规则说明

### 金额转账测试规则（通用规则）

所有涉及金额转账的用例必须覆盖最大值、最小值以及中间值。

**最大值规则**：
- 定义为钱包里的余额（用户自己的可用值）
- 需要考虑账户最低余额限制
- 需要考虑 Gas 费用预留
- 需要考虑锁定金额

**最小值规则**：
- 需要根据具体的链和代币规则来定
- 不同链可能有不同的最小转账限制
- 具体规则见下方各链规则

**中间值规则**：
- 选择余额的中间值进行测试
- 确保覆盖正常转账场景

### 硬件钱包转账测试规则

#### 硬件钱包签名流程验证规则（强制）

在硬件钱包主币转账、代币转账以及 DAPP 签名时，必须验证设备上显示的签名详情：

**盲签链处理**：
- 如果该链支持盲签，则不需要校验签名详情

**非盲签链验证项**（必须验证以下信息在硬件设备上正确显示）：
- (a) From 地址与 To 地址
- (b) 转账金额
- (c) 链的名称显示
- (d) Gas 费用等相关参数
- (e) 代币的名称（代币转账时）

**验证方式**：
- 在用例的"预期结果"中明确标注需要验证硬件设备上显示的上述信息
- 确保签名详情与交易参数一致

#### 硬件钱包转账测试用例简化规则（强制）

生成硬件钱包转账测试用例时，遵循以下简化原则：

**不覆盖内容**（已在软件钱包用例中覆盖，硬件钱包用例中不再重复）：
- 新旧账户判断（如向新账户转账 vs 向老账户转账，这部分在软件钱包用例中覆盖）
- 软件相关的金额限制（如账户最低余额、Gas 预留等详细规则）
- 激活功能（如 Token 激活、账户激活等）
- 其他软件钱包已覆盖的通用规则

**仅覆盖内容**（硬件钱包测试的三个核心内容）：
- (a) 主币转账
- (b) 代币转账（含普通代币、免 Gas 代币）
- (c) DAPP 交易（连接、签名、合约交互）

**硬件用例内容边界（强制）**：硬件用例**只写需要硬件签名的操作**，App 侧操作（连接授权 / 账户选择 / UI / 历史记录 / 输入边界等）由软件用例覆盖，硬件文档不重复。详见 `docs/qa/rules/wallet-rules.md §3.2.1`。

**测试入参和测试点**（必须覆盖）：

1. **金额参数**：
   - 最大值（账户最大可用余额）
   - 最小值（链规则允许的最小转账金额）
   - 中间值（正常转账场景）

2. **Memo 参数**（如链支持 Memo）：
   - 超长 Memo（接近或达到链的最大长度限制）
   - 短 Memo（如 1-5 个字符）
   - 不同格式的字符串（纯数字、包含字符、特殊字符等）
   - **注意**：不同链的 Memo 最大长度不同，需参考下方各链规则（如 Stellar 链 Memo 最大长度为 28 字节）

3. **硬件显示验证**：
   - 验证硬件设备上签名详情的显示是否正常
   - 验证信息清晰可读
   - 验证显示内容与交易参数一致

**预期结果要求**：
- 必须包含硬件设备上显示的签名详情（From/To 地址、转账金额、链名称、Gas 费用、代币名称、Memo 等）
- 必须包含交易提交后需要确认的信息（交易成功、余额扣除、交易记录生成等）
- 硬件确认流程验证直接整合在转账用例的预期结果中，不需要单独编写硬件确认流程章节

**用例设计原则**：
- 聚焦硬件钱包特有的验证点（签名流程、设备显示）
- 避免与软件钱包用例重复
- 保持用例简洁，突出硬件钱包测试重点
- 使用组合入参形式（金额 × Memo）进行测试，提高测试覆盖率

#### 硬件转账用例生成规则（强制）

生成硬件钱包转账测试用例时，**主币转账和代币转账各生成 3 条用例**，共 6 条用例。

**用例组合规则**（memo + 金额）：

**主币转账三条用例**：
1. **最小值 + memo（数字字符组合，最大限制）**
   - 金额：最小转账金额（参考下方各链规则）
   - Memo：数字字符组合，长度达到链的最大长度限制（参考下方各链规则，如 Stellar 链为 28 字节）
2. **最大值 + 不加 memo**
   - 金额：最大值（余额 - 账户最低余额 - Gas，参考下方各链规则）
   - Memo：不填写
3. **中间值 + 纯数字 memo**
   - 金额：中间值（余额的中间值）
   - Memo：纯数字（自动推断为对应类型，参考下方各链规则，如 Stellar 链为 MEMO_ID 类型）

**代币转账三条用例**：
1. **最小值 + memo（数字字符组合，最大限制）**
   - 金额：最小转账金额
   - Memo：数字字符组合，长度达到链的最大长度限制（参考下方各链规则，如 Stellar 链为 28 字节）
2. **最大值 + 不加 memo**
   - 金额：最大值（代币余额）
   - Memo：不填写
3. **中间值 + 纯数字 memo**
   - 金额：中间值（代币余额的中间值）
   - Memo：纯数字（自动推断为对应类型，参考下方各链规则，如 Stellar 链为 MEMO_ID 类型）

**注意事项**：
- 不同链的 Memo 最大长度不同，需参考下方各链规则（如 Stellar 链 Memo 最大长度为 28 字节）
- 不同链的 Memo 类型推断规则不同，需参考下方各链规则
- 如果链不支持 DApp 交易，不输出 DApp 相关用例（需参考下方各链规则中的硬件钱包支持情况）

## 🔗 各链转账规则

### Stellar (XLM)

#### 账户最低余额限制
- **基础保留**：账户余额不能低于 **1 XLM**
- **Token 占用**：每激活（Add Trustline）一个 Stellar Asset Token，必须锁定 **0.5 XLM**
- **示例**：账户持有 10 个 Asset Token，则有 5 XLM 被锁定不可转出

#### 主币转账（XLM）
- **向新账户转账**：金额必须 ≥ **1 XLM**（否则交易失败）
- **向老账户转账**：无最小金额限制（但需预留 Gas）
- **Gas 费用**：普通交易费用极低（默认约 100 stroops）

#### 代币转账
- **Stellar Asset Token**：
  - 接收方必须已激活（建立 Trustline）该 Token
  - 如果接收方未激活，交易应失败或前端提示无法转账
  - 精度：固定 7 位
- **Contract Token**：
  - 无需接收方激活
  - 精度需通过链上查询
  - 包含 Resource Fee，费用较高

#### Memo 规则
- **自动推断**：
  - 输入纯数字 → 自动设为 MEMO_ID 类型
  - 输入包含字符/字符串 → 自动设为 MEMO_TEXT 类型
- **不支持**：Hash 和 Return 类型的 Memo
- **最大长度**：Memo 最大长度为 **28 字节**
- **CEX 充值**：往 Binance 和 OKX 充值 XLM，务必填写 Memo

#### 硬件钱包支持
- ✅ XLM 转账：支持
- ✅ Asset Token 转账：支持
- ❌ Contract Token 转账：不支持
- ❌ Swap：不支持
- ❌ Soroban Token：不支持
- ❌ DApp 交易：不支持（Stellar 硬件钱包不支持 DApp 交易，不输出 DApp 相关用例）

### Lightning Network (BTC)

> 闪电网络是链下协议，与其他链在金额单位、历史记录字段、费用模型上存在显著差异，生成用例前必须先阅读本节。

#### 单位与精度
- **最小单位**：1 sat（satoshi），**整数**；1 BTC = 100,000,000 sats
- **单笔上限**：100,000,000 sats（1 BTC）；超出提示「金额不得超过 100000000 sats」
- **法币换算**：法币金额 ↔ sats 可双向切换；法币过小时可能显示为 0
- **法币输入 → sats 必须四舍五入为整数**（强制）：
  - 输入法币金额时，换算出的 sats **必须显示为整数**，不得出现小数
  - 若显示非整数 sats（如 `123.6 sats`），交易将失败（链下协议不支持非整数 sats）
  - 确认页、历史记录中的金额同样必须为整数 sats

#### 金额与余额规则
- **指定金额发票**（BOLT11 携带金额）：
  - 金额自动填入，**输入框不可修改**
  - 若发票金额 + 1% 浮动费用 > 账户余额，禁止提交，提示「余额不足！请至少预留 invoice 金额的 1% 来应对潜在的交易费浮动」
- **未指定金额发票 / LN Address / LNURL**：
  - 金额 = 0 或空：禁止提交，提示输入金额
  - 金额 > 1 BTC 或超过可用余额：禁止提交
  - 金额最小值 1 sat；点击「Max」按钮 = 可用余额

#### 收款地址类型（3 种）
| 类型 | 格式 | 示例 |
| --- | --- | --- |
| 发票（BOLT11） | `lnbc...` / `lntb...` | `lnbc300u1pj2...` |
| LN Address | `user@domain` | `shortmen02@walletofsatoshi.com` |
| LNURL | `lnurl1...` | `lnurl1dp68gurn8...` |

#### 发票描述（Memo / 备注）
- **最大长度**：40 字符（含空格、换行、特殊字符、Emoji）
- **超长**：禁止提交，提示「描述不得超过 40 字符」
- **未填写默认值**：`OneKey Invoice`
- **对端携带**：由开票方写入，收款方不可改

#### 网络费规则
- 交易费 = 0 或由路由决定（发票携带）
- **不可修改、不可切档**（无 Fast / Standard / Slow）
- 每次交易金额可能不同（不固定）

#### 历史记录特性（与其他链的关键差异）
| 字段 | 闪电网络 | 其他链 |
| --- | --- | --- |
| 交易哈希跳转 | **无跳转**（闪电网络无区块浏览器） | 可跳转区块浏览器 |
| 交易对象 From/To | **仅展示当前账户** | 同时展示双方 |
| 网络费调节 | **不可修改** | 通常可切档 |
| 网络标识 | `Lightning` / `闪电网络` | 各链独立名称 |
| 交易完成时长 | 秒级 | 分钟级 |
| 主币余额 = 0 发送前置拦截 | **不生效** | 单链视图下生效 |

#### 异常发票处理
| 发票状态 | 预期提示 |
| --- | --- |
| 已过期 | 地址无效 |
| 已被支付过 | 发票已支付 |
| 格式错误（截断 / 缺前缀） | 地址无效 |

#### 账户与选择地址
- **「我的账户」Tab 不支持闪电网络**：发送流程中选择地址弹窗仅显示「最近」Tab（参考 `wallet-rules.md §5.05`）
- **账户间数据隔离**：多个闪电账户的余额、历史记录独立，切换账户后刷新为对应账户数据

#### HW 钱包支持
- ✅ 主币转账（发票 / LN Address / LNURL）：支持
- ✅ dApp 支付（LApp Deposit 等发送类交易）：支持
- ❌ 代币转账：闪电网络无代币概念

#### HW 钱包签名范围（生成 HW 用例时必须遵守）
| 操作 | 是否需要 HW 签名 | 说明 |
| --- | --- | --- |
| 发送（发票 / LN Address / LNURL） | ✅ 需要 | HW 设备按「发送 / Deposit 签名页字段」显示（见下表） |
| 接收（生成发票） | ❌ 不需要 | App 侧生成 |
| dApp 连接授权（App 弹出授权弹窗） | ❌ 不需要 | App 内完成，不涉及 HW |
| dApp 支付（LApp Deposit） | ✅ 需要 | 属发送类交易，HW 签名页字段同发送 |
| dApp 赎回（LApp Withdraw） | ❌ 不需要 | 属接收类 |
| dApp 授权登录（LNURL-auth：Alby / WebLN / Nostr Wallet Connect） | ✅ 需要（HW 钱包场景） | HW 钱包需 HW 签名 linkingKey；HD 钱包在 App 内完成不涉及 HW。HW 签名页字段见下表「LNURL 授权签名页字段」 |
| 账户选择 / App UI 操作 | ❌ 不需要 | 与 HD 钱包一致 |
| 通道开启 / 关闭 | ✅ 需要 | 涉及链上交易 |

#### HW 设备型号签名页字段差异（所有 HW 用例默认回归所有型号）

| 型号 | 发送 / Deposit 签名页字段 | LNURL 授权签名页字段 |
| --- | --- | --- |
| Mini | 发送方地址；消息内容（创建时间 / 过期时间 / 发票等） | 标题 = `LNURL 授权`；域名；展示数据 |
| Classic / Classic 1S | 标题 = `签署 Bitcoin 消息`；发送方地址；消息内容（创建时间 / 过期时间 / 发票等） | 标题 = `批准 LNURL 授权`；域名；展示数据 |
| Touch / Pro | 标题 = `签署 Bitcoin 消息`；消息内容（创建时间 / 过期时间 / 发票等）；发送方地址 | 展示消息数据；域名 |

> 断言口径：每行列出的字段均需在 HW 屏上可见，内容与 App 侧推送一致。HW 用例无需为每个型号拆独立用例，在「预期结果」中以「按设备型号字段差异表逐型号校验」统一描述即可。

#### HW 钱包用例范围约束
- HW 钱包用例**仅覆盖需要 HW 签名的操作**，其他 App 侧操作（连接授权、UI、历史记录、账户选择等）由 HD 钱包用例覆盖，HW 文档**不重复描述**
- HW 用例预期聚焦签名屏（发送方 / 金额 / 链名 / **发票描述** / 网络费）+ 上链 + 余额
- 不需要在 HW 用例中验证 App 侧的确认页字段、历史记录详情、UI 无异常等

#### dApp（LApp）交互
- **连接协议**：Nostr Wallet Connect / Alby / WebLN
- **典型 dApp**：LNBlackjack（Deposit / Withdraw）、LNMarkets（LNURL-auth 授权登录）
- **支付时确认页必须显示**：dApp Origin、发票描述、金额、手续费、发送方 = 自己
- **LNURL 授权登录（LNURL-auth）**：
  - HD 钱包：在 App 授权页内完成（无设备交互）
  - HW 钱包：App 推送签名请求到 HW，HW 按「HW 设备型号签名页字段差异」表展示域名 / 数据 / 标题后确认
  - 站点域名断言以第三方服务实际下发为准（如 LNMarkets = `api.lnmarkets.com`）

## 📝 规则维护指南

### 如何添加新链规则

1. **收集规则信息**：
   - 账户最低余额限制
   - 主币最小转账金额
   - 代币转账规则（是否需要激活、精度等）
   - Gas 费用规则
   - 特殊规则（如 Memo、Tag 等）
   - 硬件钱包支持情况

2. **格式要求**：
   - 使用清晰的标题和子标题
   - 使用表格展示对比信息
   - 使用代码块展示示例
   - 标注支持/不支持状态（✅/❌）

3. **验证规则**：
   - 规则必须经过实际测试验证
   - 如有疑问，标注"待验证"或"需确认"

### 如何更新现有规则

1. **发现规则变更**：
   - 在测试过程中发现规则与文档不一致
   - 收到产品/开发通知规则变更
   - 链本身规则更新

2. **更新文档**：
   - 直接修改对应链的规则部分
   - 在变更记录中记录更新时间和原因

3. **通知相关方**：
   - 如规则变更影响现有测试用例，需同步更新用例

### Cosmos 系（Cosmos SDK / IBC）

> 适用于 Akash、Cosmos Hub、Cronos POS Chain（原 Crypto.org）、Fetch.ai、Juno、Osmosis、Secret Network、Celestia、Babylon、Noble 等基于 Cosmos SDK 的网络（以 OneKey 已接入列表为准）。

#### Memo / 备注

- **最大长度**：**512 字节**（按 UTF-8 编码占用字节数计算）。
- **字符类型**：允许空备注；允许纯数字、字母、中文、Emoji、混合文本等（不限制字符形式）。
- **超限处理**：超过 512 字节时**禁止提交**。
- **边界测试**：必须包含「恰好 512 字节」与「513 字节（或超过上限）」两类用例。

#### 金额与精度

- **最大值**：当前账户在该网络上的**余额上限**（最大可转金额；需预留本链 Gas，具体预留以产品/链实现为准）。
- **最小值**：该币种支持的**最小精度单位**（由代币 decimals 决定）。
- **禁止 0**：金额 = 0 时**禁止提交**。
- **代币类型**：需同时覆盖**网络主币**与 **IBC / 原生稳定币等代币**（如 Osmosis 上 ATOM、Noble 上 USDC）。
- **IBC 代币**：支持 IBC 代币转账（链上资产标识常见为 `ibc/<hash>`），用例需覆盖至少 1 个 IBC 代币。

#### Gas / 费用

- 预览页需支持**多档网络费用**切换；切换后预估费用与广播前展示一致（与链上实际扣费允许合理波动，须在区块浏览器可核对）。
