## 2. 账户聚合规则（EVM 跨链 / UTXO xpub）

### 2.1 EVM 跨链聚合与链信息展示

| 优先级 | 场景 | 操作步骤 | 预期结果 |
| --- | --- | --- | --- |
| P1 | 同一 EVM 地址跨链记录 | 打开 EVM 网络下的「最近转账」Tab | 确认前端列表无重复项（去重逻辑已由接口自动化保障） |
| P1 | EVM 聚合结果中每条记录带有 `networkId` | 打开 EVM 网络下的「最近转账」Tab | 列表项显示对应链名称徽标/文案，与最近一次转账所在网络一致 |
| P1 | 非 EVM 链存在最近转账记录 | 打开非 EVM 网络下的「最近转账」Tab | 仅显示当前链结果，不混入其他链地址 |

### 2.2 BTC / LTC 扩展公钥（xpub）聚合

> **核心规则**：**仅 BTC / LTC** 的最近转账按扩展公钥统计，同一账户下所有派生路径（Taproot / Native SegWit / Nested SegWit / Legacy）以及历史使用过的旧地址产生的转账**都会**进入列表；只要转账发自当前账户的某个派生地址，就会出现在最近转账里。
> **DOGE / BCH** 按地址聚合（与 Tron 同规则），不走 xpub 聚合。

| 优先级 | 场景 | 操作步骤 | 预期结果 |
| --- | --- | --- | --- |
| ❗️❗️P0❗️❗️ | BTC / LTC 多地址类型聚合（参数化） | 1. 使用曾用多种地址类型（Taproot / Native SegWit / Nested SegWit / Legacy）收发过币的账户<br>2. 进入发送页打开「最近转账」Tab | 所有地址类型下产生的转账对手地址都出现在列表中，不因当前选中的地址类型不同而丢失<br>**参数集**（每条都要验证）：BTC / LTC |
| ❗️❗️P0❗️❗️ | BTC / LTC 历史（旧）地址的转账保留（参数化） | 1. 账户曾用历史地址 `addrOld` 发起过转账<br>2. 当前收款地址已切换到 `addrNew`<br>3. 打开「最近转账」Tab | `addrOld` 发起的转账对手仍然出现在列表中，不因地址更换而消失<br>**参数集**（每条都要验证）：BTC / LTC |
| P1 | 不同账户（不同 xpub）不串数据 | BTC / LTC 链下有账户 A、账户 B 两个独立 HD 账户，分别有不同的转账历史 | 账户 A 的「最近转账」只显示 A 的 xpub 下派生地址产生的记录，不混入 B 的记录 |
| P1 | DOGE / BCH 按地址聚合验证 | 在 DOGE / BCH 账户打开「最近转账」Tab | 列表按地址维度展示，与 Tron 规则一致；不应混入同 HD 其它地址的记录 |

## 8. 本地历史路径过滤人工抽查

> 目的：验证客户端对本地历史的过滤规则（合约交互、0 金额）和账户隔离。
> 抽查方式：**必须用户自己在本地发起过交易**（观察地址无本地历史），以下场景需人工操作钱包完成一笔交易后再验证。

| 优先级 | 场景 | 操作步骤 | 预期结果 |
| --- | --- | --- | --- |
| ❗️❗️P0❗️❗️ | 非 API 链 dapp swap 合约交互过滤（参数化） | 1. 在目标链账户下，用对应 dapp 发起一笔 swap<br>2. 返回发送页，打开「最近转账」Tab | 该 swap 的合约 / 程序交互地址**不**出现在列表中<br>**参数集**（每条都要验证）：<br>- Aptos（任一 dapp）<br>- Sui（任一 dapp）<br>- Cardano（任一 dapp）<br>- Solana（Jupiter / Raydium 等；**API 上线后需迁移到 Section 7 API 抽查**） |
| P0 | 本地历史 0 金额过滤 | 在非 API 支持链发起一笔 0 金额转账，返回「最近转账」Tab | 该 0 金额记录**不**出现在列表中 |
| ❗️❗️P0❗️❗️ | 本地历史账户隔离 | 1. 账户 A 在非 API 链发起转账<br>2. 切到同链账户 B 的「最近转账」Tab | 不显示账户 A 的本地历史记录 |
| P1 | 本地历史普通转账保留 | 在非 API 链完成一笔普通外部地址转账 | 新地址出现在「最近转账」列表最前 |

## 9. 跨钱包类型数据一致性人工抽查

> 目的：验证同一个闪电网络账户在 HD 钱包 / HW 钱包 / 观察地址下导入，最近转账中的 **LNURL 地址**展示一致（不拼错、不截断、不显示原始 pubkey）。
> 抽查链：**闪电网络（Lightning）** —— 底层绑定 BTC 账户，可在 HD 钱包 / HW 钱包下同时创建或导入。
> **不做的断言**：不要求两份列表条数 / 顺序 / memo / tag 完全一致（最近转账来源、缓存、本地历史可能差异），只抽查 LNURL 地址字符串一致。

| 优先级 | 场景 | 操作步骤 | 预期结果 |
| --- | --- | --- | --- |
| ❗️❗️P0❗️❗️ | HD 钱包 vs HW 钱包 LNURL 地址一致性 | 1. HD 钱包下导入 / 生成闪电网络账户，取一条最近转账的 LNURL 地址<br>2. HW 钱包下以**同一助记词 / 同一账户**导入闪电网络账户<br>3. 找到同一对手的最近转账记录，对比 LNURL 地址字符串 | 同一对手在两侧展示的 LNURL 地址**字符串完全一致**（字符、长度、大小写一致，未截断，未退化为原始 pubkey） |
