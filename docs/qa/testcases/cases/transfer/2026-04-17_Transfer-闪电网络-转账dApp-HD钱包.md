# Transfer - 闪电网络 - 转账/dApp-HD钱包

> 需求文档：`docs/qa/requirements/Wallet-闪电网络.md`
> 规则文档：`docs/qa/rules/transfer-chain-rules.md` §「Lightning Network (BTC)」、`docs/qa/rules/wallet-rules.md`
> 关联用例：`docs/qa/testcases/cases/transfer/2026-04-17_Transfer-闪电网络-转账dApp-HW钱包.md`
> 测试端：iOS / Android / Desktop / Extension / Web（以产品支持为准）
> 测试范围：指定金额发票 / 未指定金额发票 / LN Address / LNURL 发送、生成发票（接收）、LApp（LNBlackjack / LNMarkets）连接 · 支付 · 授权登录、断网恢复
> 金额单位：sats（整数），1 BTC = 100,000,000 sats；发票描述 ≤ 40 字符，默认 `OneKey Invoice`；网络费不可修改；交易哈希可复制、无区块浏览器跳转

3. 闪电网络地址：`shortmen02@walletofsatoshi.com`（LN Address）、`lnurl1dp68gurn8ghj7ampd3kx2ar0veekzar0wd5xjtnrdakj7tnhv4kxctttdehhwm30d3h82unvwqhhx6r0wf6x6etwxqeq3p2554`（LNURL，失效则用 `batchNewUrl` 脚本重新生成）

## 1. 发送 - 成功路径

**通用操作步骤**：粘贴 / 输入收款信息 →（必要时）输入金额或点「最大值」→「确认发送」→ 成功后核对提示、loading 与历史详情。

| 优先级 | 场景 | 操作步骤 | 预期结果 |
| --- | --- | --- | --- |
| ❗️❗️P0❗️❗️ | 指定金额发票 · 最小额 | 粘贴最小合法金额指定金额发票（如 1 sat，≤ 余额 × 99%）→「确认发送」 | 1. 金额输入框禁用、不显示法币<br>2. 网络费显示且不可修改<br>3. 「转账成功」；各 loading ≤ 2s；无 UI 错位、闪屏、非预期错误<br>4. 历史详情：金额 = 实际支付 sats（整数）= 发票载明金额；状态 = 成功；发票描述 = 随发票 / 默认 `OneKey Invoice`；网络费用 = 0 sats；交易哈希可复制、无跳转区块浏览器；网络 = Lightning |
| ❗️❗️P0❗️❗️ | 未指定金额发票 · Max | 粘贴未指定金额发票 → 点「最大值」→「确认发送」 | 1. 金额输入框 = 当前可用余额<br>2. 其余**同上**，历史详情金额 = 该笔实际扣款 sats |
| ❗️❗️P0❗️❗️ | LN Address · 中间值 | 输入 `shortmen02@walletofsatoshi.com` → 解析进入金额页 → 输入中间值（如 `1000` sats）→ 切换 sats ↔ 法币 →（可选填描述 ≤ 40 字符）→「确认发送」 | 1. 地址解析正确；金额可编辑；`sats = round(法币 ÷ BTC 价 × 1e8)`，误差 ≤ 1 sat<br>2. 其余**同上**，发票描述 = 随填写 / 默认 `OneKey Invoice`；接收方展示与 LN Address 一致 |
| ❗️❗️P0❗️❗️ | LNURL · 中间值 | 粘贴 `lnurl1dp68gurn8ghj7ampd3kx2ar0veekzar0wd5xjtnrdakj7tnhv4kxctttdehhwm30d3h82unvwqhhx6r0wf6x6etwxqeq3p2554` → 解析进入金额页 → 同「LN Address · 中间值」→「确认发送」 | **同上**，接收方展示与 LNURL 解析结果一致 |

## 产品体验建议（QA 视角）

> 以下建议基于测试体验，非功能缺陷，供产品 / 设计参考。

### 安全风险

- 【建议】**发票描述区分来源**：LApp 支付时发票描述由 dApp 写入（如 `LNBlackjack.com`），与用户本端生成的描述混同展示。建议在历史详情中区分「对端发起」与「本端填写」，避免钓鱼站伪造熟悉的描述文案误导用户。

### 易用性

- 【建议】**无浏览器跳转的说明**：哈希行保持对齐其他链的交互心智（可点击 / 指针样式）但实际无跳转，用户易产生「点击无反应」疑虑。建议在哈希右侧直接用「仅可复制 · 闪电网络无区块浏览器」的 icon / 轻提示对齐预期。
- 【建议】**1% 预留提示拆分**：「余额不足！请至少预留 invoice 金额的 1% 来应对潜在的交易费浮动」单行信息密度过高，建议拆为两行：`余额不足` + `请预留 invoice 金额 1% 作为路由费浮动`。

### 操作效率

- 【建议】**Max 按钮文案本地化**：当前未指定金额发票 / LN Address / LNURL 路径下的「最大值」按钮建议与其他链统一写作「Max」，避免同一 App 内两种说法并存。
